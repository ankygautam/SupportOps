# SupportOps Screenshot Plan

## 1. Demo entry

- Route: `/demo`
- Role: public
- Show: role cards, guided explanation, quick sign-in buttons, launch checklist link
- Goal: establish product context and polished demo access

## 2. Dashboard

- Route: `/dashboard`
- Role: `Admin` or `Team Lead`
- Show: top summary cards, recent activity, assigned work, incident/SLA context
- Goal: capture the overall operations-control-room feel

## 3. Tickets list

- Route: `/tickets`
- Role: `Team Lead`
- Show: saved view chips, filters, summary cards, populated table with mixed priorities and SLA states
- Goal: show the strongest queue-management screen

## 4. Ticket detail

- Route: `/tickets/:id`
- Role: `Team Lead`
- Show: activity timeline, comments/internal notes, SLA card, assignment workflow, escalation controls
- Goal: demonstrate workflow depth and operational realism

## 5. Incidents

- Route: `/incidents`
- Role: `Team Lead`
- Show: summary cards, incident table, active detail drawer if possible
- Goal: show how support and incident management connect

## 6. Analytics

- Route: `/analytics`
- Role: `Admin`
- Show: top metrics, team performance, trends, insights panels
- Goal: capture leadership-facing product depth

## 7. Team or settings

- Route: `/team` or `/settings`
- Role: `Admin`
- Show: staffing controls or role-aware preference sections
- Goal: reinforce that the product goes beyond dashboards

## Capture notes

- Use desktop width first.
- Keep the left sidebar, top header, and main content area visible.
- Prefer populated states over empty states for hero screenshots.
- If recording a walkthrough, follow the same order as the demo guide.
